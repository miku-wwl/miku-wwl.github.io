# Practical SQL optimization in day-to-day backend work

SQL optimization should start with measurement, not taste. A query that looks ugly may be fast, and a query that looks clean may scan millions of rows. The database only cares about data distribution, indexes, plan choice, and runtime cost.

My first step is always to identify the exact slow query and reproduce it with realistic parameters.

## Read the execution plan

`EXPLAIN` is the entry point. I look for:

- Full table scans on large tables.
- Bad row estimates.
- Missing or unused indexes.
- Temporary tables.
- Filesort operations.
- Expensive nested loops.
- Large result sets before filtering.

The plan should be read together with table size, index cardinality, and the actual parameter values. A query can be fast for one tenant and slow for another if the data distribution is uneven.

## Index for the access pattern

An index is not just a column list. It is a data structure optimized for a specific access pattern.

Useful rules:

- Put equality filters before range filters in composite indexes.
- Match index order with `ORDER BY` when possible.
- Avoid functions on indexed columns in the predicate.
- Do not create overlapping indexes without a reason.
- Watch selectivity; low-cardinality indexes are often less useful alone.

The goal is not to add more indexes. The goal is to add the right index and remove indexes that only slow writes.

## Pagination matters

Offset pagination becomes expensive when the offset is large:

```sql
select * from orders order by id limit 20 offset 100000;
```

The database may still need to walk through the skipped rows. Keyset pagination is usually better:

```sql
select * from orders
where id > ?
order by id
limit 20;
```

Keyset pagination needs a stable sort key and slightly different API design, but it performs much better for deep pages.

## Avoid fetching too much

Many slow queries are slow because they return more data than the caller needs.

Common fixes:

- Select only required columns.
- Add server-side filters.
- Split detail queries from list queries.
- Limit result size.
- Avoid large `IN` lists generated by application code.
- Batch secondary lookups deliberately.

Large result sets also create application pressure: serialization, network transfer, memory allocation, and client rendering.

## Watch lock contention

SQL performance is not only read latency. Writes can be slow because of locks.

I check:

- Transaction duration.
- Rows locked by an update.
- Whether indexes support update predicates.
- Deadlock logs.
- Foreign key checks.
- Hot rows or counters.

Long transactions are especially dangerous because they hold locks while application code does unrelated work. Keep transactions small and move remote calls outside the transaction boundary.

## When schema is the fix

Sometimes query tuning is not enough. The schema may not match the workload.

Possible schema-level fixes include:

- Add a summary table.
- Denormalize carefully for read-heavy paths.
- Partition or archive historical data.
- Split hot and cold columns.
- Introduce a search engine for text search.
- Use a materialized view or precomputed projection.

These changes have operational cost, so they should be justified by real load and monitored after release.

## My review checklist

Before calling a SQL optimization done, I want to know:

- What was the original latency and plan?
- What changed?
- What is the new plan?
- Does the fix work for realistic data distribution?
- Does it increase write cost?
- Is there a regression test or dashboard?

Good SQL optimization is not clever SQL. It is disciplined feedback between workload, data model, and execution plan.
