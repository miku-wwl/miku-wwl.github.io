# Snowflake IDs: sortable distributed identifiers

Snowflake-style IDs are 64-bit identifiers designed for distributed systems. They are compact, roughly time-ordered, and can be generated without a central database sequence.

A typical layout is:

```text
sign bit | timestamp | worker id | sequence
```

The exact number of bits can vary by implementation.

## Why they are useful

Distributed services often need IDs before writing to the database. A database auto-increment key can become a bottleneck or make sharding harder.

Snowflake IDs provide:

- Local generation.
- High throughput.
- Time ordering.
- Compact storage compared with UUID strings.
- A place to encode worker or shard information.

The time ordering is especially useful for logs, orders, and event records.

## Sequence within a millisecond

The sequence part handles multiple IDs generated by the same worker during the same millisecond. If the sequence space is exhausted, the generator usually waits until the next millisecond.

This gives predictable throughput per worker. More workers can increase total throughput, as long as worker IDs are assigned safely.

## Worker ID assignment

Every generator must have a unique worker ID. If two nodes use the same worker ID at the same time, duplicate IDs can be generated.

Worker IDs can be assigned by:

- Static configuration.
- Kubernetes pod identity.
- A coordination service.
- A startup registration process.

The assignment mechanism should handle restarts and avoid accidental reuse.

## Clock rollback

Clock rollback is the most important failure mode. If the system clock moves backward, a worker may generate IDs with timestamps it has already used.

Mitigations include:

- Refuse to generate IDs until time catches up.
- Use a small tolerated rollback window.
- Alert on clock movement.
- Use monotonic time logic where possible.
- Keep NTP configuration healthy.

Clock issues are operational issues, not only code issues.

## Tradeoffs

Snowflake IDs are not globally sequential. They are only roughly ordered by time. They can also reveal timestamp information, which may or may not be acceptable.

For public identifiers, teams sometimes use a separate opaque ID to avoid exposing creation time or infrastructure details.

The design is strong when the system needs fast local ID generation and operational teams understand the clock and worker identity assumptions.
